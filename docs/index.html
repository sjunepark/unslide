This page documents the current V1 implementation. The accepted V2 direction replaces required A4/chrome source and copy-in adoption with a headless artifact protocol, versioned tooling, fully report-owned design, and HTML-first PDF export. See the live plan before starting new work.

What Unslide is

V1 is a small, repository-local foundation for deliberately composed reports—not a presentation app, document flow engine, or general component library.

Explicit pages

Page membership stays visible in source. Content never flows to another page automatically.

Ordinary code

Typed data, React server rendering, semantic HTML, and normal CSS. No proprietary template language.

Rendered truth

Generated page images expose wrapping, font metrics, balance, and overflow before delivery.

Standalone output

The final HTML embeds its CSS and opens without Node, Playwright, a server, or a network request.

i

The author owns content fit. If content overflows, revise the source, typography, layout, or page allocation. V1 deliberately has no automatic repair path.

Run the project

The shortest path from a clean checkout to a browser-rendered report.

Terminal 
pnpm install --frozen-lockfile
pnpm exec playwright install chromium
pnpm run render:spike
pnpm run capture:spike
open artifacts/spike/report.html

The render command writes a standalone three-page report to artifacts/spike/report.html. Capture writes one PNG per explicit page under .tmp/captures/spike/.

Author a report

Keep calculations in caller-owned code. Pass ready-to-display data into a report component, compose a known page list, then write the static artifact.

  1. Model display-ready data

    Use an ordinary TypeScript type and object. Format domain values before they reach the shared foundation.

  2. Compose explicit pages

    Render each page with its final number and total. Reuse chrome when the report calls for it; omit it intentionally on covers.

  3. Own report-specific CSS

    Layouts, tables, figures, typography, and visual language stay beside the report. The foundation owns only fixed geometry and chrome regions.

  4. Render, capture, inspect, revise

    Inspect every page image after content or layout changes. The source is editable intent; the browser output is visual truth.

report.tsx 
import { Page } from "../../unslide/page.js";
import type { ReportData } from "./data.js";

export function Report({ data }: { data: ReportData }) {
  const total = 2;
  const chrome = {
    headerLeft: data.company,
    headerRight: "Operating review",
    footerLeft: "Confidential",
  };

  return (
    <main className="report">
      <Page number={1} total={total} className="cover">
        <h1>{data.title}</h1>
      </Page>
      <Page number={2} total={total} chrome={chrome}>
        <h2>{data.headline}</h2>
      </Page>
    </main>
  );
}

V1 source adoption

This is the verified V1 path only. V2 will package stable build, validation, capture, and PDF behavior while leaving every visual decision in report-owned source.

V1 repository structure

Separate shared mechanics from report-owned data and composition. A new report should not require changes under src/unslide/.

src/unslide/ ├── foundation.css # page geometry and print behavior ├── page.tsx # page, chrome, and numbering └── render.tsx # standalone HTML writer src/reports/my-report/ ├── data.ts # caller-owned typed values ├── report.tsx # explicit page composition ├── render.tsx # report entry point └── styles.css # report-specific visual system scripts/ └── capture.ts # development-only Chromium adapter artifacts/my-report/ └── report.html # standalone delivery output .tmp/captures/my-report/ └── page-01.png # disposable inspection output

The renderer reads foundation.css next to itself, combines it with report CSS, renders the TSX body to static markup, and embeds both in one HTML file.

The V1 interface

Two exports cover the reusable runtime. Capture remains a shell-driven development adapter.

<Page>

src/unslide/page.tsx

Creates one fixed A4 landscape page with a content region and optional repeated chrome.

number: number · total: number · chrome?: PageChrome · className?: string · children

PageChrome

interface

Provides headerLeft, headerRight, and footerLeft as React nodes. Page position is rendered at footer right.

writeReportHtml()

src/unslide/render.tsx

Writes the standalone document shell. Options are title, body, reportStyles, and outputPath.

capture.ts

development command

Usage: tsx scripts/capture.ts <report.html> <output-directory>. It validates and captures [data-unslide-page] elements, writing one PNG per page in isolated Chromium.

Rendered inspection is part of authoring

Do not approve a report from TSX or HTML source alone. Font metrics, line wrapping, table width, and visual balance exist only after layout.

  1. Regenerate the artifact

    Run the report’s render command after changing data, wording, structure, or CSS.

  2. Capture every page

    The capture command removes stale page-*.png files from its target and writes a fresh ordered set.

  3. Inspect at page scale

    Check content fit, clipped or overlapping elements, legibility, alignment, charts, tables, bilingual text, and intentional page breaks.

  4. Correct the source

    Shorten content, adjust layout, tune type, split a table manually, or add an explicit page. Then repeat the loop.

✓

Canonical preview: Playwright-managed Chromium. The delivered HTML can open in a modern browser, but V1 does not promise cross-browser pixel parity.

Know the V1 boundary

This is the implemented V1 ownership split. The accepted V2 design moves geometry and repeated material out of the runtime; follow the live plan for migration status.

Unslide owns

  • Fixed A4 landscape page geometry
  • Optional chrome and page numbering
  • Standalone static HTML output
  • Screen and print page separation
  • Isolated Chromium page capture

Callers or future scope own

  • Business calculations and domain models
  • Automatic pagination or fit repair
  • A visual editor or presentation controls
  • A large report component catalogue
  • Stable package distribution or multi-renderer support

Troubleshooting

No report pages found

The capture script looks for elements with a nonempty, unique data-unslide-page ID. The V1 Page component supplies this marker; arbitrary HTML may supply it directly.

Chromium is missing

Run pnpm exec playwright install chromium. Playwright is needed only for capture, not to open the generated artifact.

TSX imports fail

Use the committed NodeNext settings and keep .js extensions in relative TypeScript imports, as shown by the working reports.

Content leaves the page

This is visible by design. Change report content, page allocation, or report CSS. Do not hide overflow or add automatic shrinking.

Output needs a server

Assets or styles were probably linked externally. Keep required CSS and assets local or embedded so the generated report remains standalone.
