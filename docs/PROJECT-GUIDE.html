Big picture

Unslide replaces binary presentation authoring with ordinary TypeScript, React, HTML, and CSS for reports whose pages are composed deliberately.

Unslide is a strict delivery pipeline around report-owned HTML—not a presentation framework and not a design system.

What it does

Builds a complete standalone HTML document, validates a tiny page/readiness protocol, captures real browser output, derives a validated PDF, and rasterizes the actual PDF for inspection.

What it deliberately does not do

It does not paginate, move overflowing content, calculate business metrics, prescribe page chrome, inject a theme, provide a visual editor, or hide report design in configuration.

Who owns what

This three-way boundary explains almost every architectural choice. Select an owner to see its responsibilities.

Report source

The visual author. It produces the complete document that Unslide will deliver.

  • Complete HTML tree and semantics
  • Explicit page sequence
  • Geometry and content fit
  • Typography and colors
  • Fonts and local assets
  • Headers, footers, numbering
  • Screen presentation
  • Print CSS and page breaks

The rule: lifecycle mechanics belong in versioned tooling; every visual decision remains readable and editable in the report’s own source.

How it works

Click through the pipeline. HTML is the center of gravity: every downstream command consumes it or an artifact derived from it.

Standalone HTML is canonical

The build step serializes the caller’s complete React document, rejects unresolved dependencies, and publishes one self-contained file.

  • React and Playwright are not needed to open the result.
  • No Unslide wrapper, CSS reset, geometry, or chrome is injected.
  • Protocol validation happens when browser-based commands consume the HTML.

Important: build does not automatically run capture or export.

Key decisions

These are accepted constraints, not unfinished conveniences. Future work should revisit them only with the evidence each record asks for.

D1

Explicit fixed pages

Page membership is authored. Long content and large tables are split manually after rendered inspection; Unslide never redistributes content.

D2

Rendered preview

Source is editable intent; isolated Chromium output is visual truth. Review does not depend on a maintainer’s personal browser session.

D3

Headless protocol

A nonvisual HTML marker separates stable tooling from report design. React is proven authoring, but HTML is the durable seam.

D4

HTML-first PDF

PDF is printed from canonical HTML and inspected from the produced PDF. There is no screenshot-to-PDF or parallel layout path.

Reviewer and maintenance focus

Use these questions to judge future changes against the project’s actual design.

  • Does the change preserve HTML as the one canonical artifact?
  • Does visual policy remain entirely in report-owned source?
  • Does config stay operational rather than becoming a design schema?
  • Does browser and PDF behavior remain behind private seams?
  • Are prior good artifacts preserved when generation fails?
  • Were both screen-media HTML and print-media PDF inspected?
  • Is a new public abstraction backed by two proven implementations?

Know the support boundary

  • Verified environment: macOS arm64, Node 24.15, pnpm 11.12.
  • One common page geometry per PDF.
  • No automatic pagination or overflow repair.
  • No async-ready author signal in protocol v1.
  • No PDF/A, signing, encryption, forms, or alternate engines.
  • Tagged output is evidence—not PDF/UA or WCAG certification.
  • Version 0.1.0 is a private pre-stable contract.

Important confusion points

These distinctions save the most time during a first review.

Does “standalone HTML” allow external resources?

No. Report code explicitly inlines selected local CSS, images, SVG, and font assets. The renderer rejects unresolved file or network dependencies.

Does a passing protocol check mean the page looks good?

No. The protocol checks page identity and static readiness. It does not judge layout, measure overflow, or approve visual quality. Every page image still needs inspection.

Does `export` rebuild the HTML first?

No. Commands are composable and explicit. `capture` and `export` consume existing HTML; `inspect-pdf` consumes an existing PDF. Rebuild upstream artifacts when source changes.

Is the report-local `Page` component part of Unslide?

No. The operating review owns that helper because its headers, footers, numbering, and A4 geometry are specific to that report. The spike proves those concepts are optional.

Why does PRODUCT.md discuss V1 limits if PDF exists now?

V1 is historical baseline evidence. Accepted V2 added the headless protocol, package adoption, and validated HTML-first PDF workflow. Current behavior is summarized by PLAN.md and ARCHITECTURE.md.

What to remember

If these three ideas stay clear, the rest of the codebase becomes predictable.

HTML is canonical.

Capture and PDF export consume the same standalone artifact. PDF inspection consumes the actual PDF.

Design stays upstream.

Caller owns calculations; report owns the complete document and fit; Unslide owns mechanics.

Rendered evidence closes the loop.

A valid source file is not a visually approved report. Inspect every target-native page.
