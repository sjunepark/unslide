# Product

## Product Idea

Unslide replaces manual static-report authoring in PowerPoint-like tools with
plain-text code and HTML. It is aimed at reports whose pages are deliberately
composed rather than automatically typeset.

The name reflects what it is not: it does not treat pages as animated slides,
and it does not reproduce presentation software. A page is simply a fixed
canvas in a static document.

## Problem

Static reports built in presentation software have several weaknesses:

- Values must be copied or linked from calculation tools.
- Links between workbooks and presentation files are fragile.
- Binary authoring formats are awkward for coding agents to read and edit.
- Repeated formatting changes are manual and difficult to review as diffs.
- Report logic, data, wording, and visual placement become entangled.

HTML already provides text, tables, images, vector graphics, typography, and
layout. Code supplies reusable structure, ordinary data flow, and agent-legible
source. Unslide combines the two without trying to become a general document
processor.

## Thesis

For deliberately designed static reports, code is a better authoring medium
than a binary slide file.

The source should be easy for people and agents to inspect, values should come
directly from code, and the final appearance should be verified through the
same browser rendering that produces the report.

Rendering remains necessary. This is true for HTML, PPTX, DOCX, PDF-first
libraries, SVG, and typesetting systems alike. HTML is attractive because both
the editable source and the rendered document remain inspectable with ordinary
web tooling.

## Primary User

V1 serves a technical professional who:

- already produces calculations or analysis in code;
- wants to build a polished, repeatable static report from those results;
- works with coding agents as report authors and reviewers; and
- is comfortable explicitly deciding what belongs on each page.

The first user is the project author. General distribution and nontechnical
authoring are later questions.

## Core Job

Given report source and ordinary program data, produce a fixed-page HTML report
that:

- opens locally in a browser;
- looks like a deliberate report rather than an application screen;
- has stable page dimensions and content regions;
- can include repeated headers, footers, and page numbers;
- can be inspected page by page as rendered images; and
- can be edited and regenerated by a coding agent without manipulating a
  binary document.

## V1 Scope

V1 proves only the core authoring loop:

1. Write a report in typed, plain-text source.
2. Pass ordinary values and collections into that source.
3. Compose a known number of fixed pages explicitly.
4. Generate static HTML.
5. Render the HTML in isolated headless Chromium.
6. Capture page images that a human or agent can inspect.
7. Revise the source and repeat.

The report author owns content fit. If a page overflows, the author or agent
changes the page composition. The runtime does not automatically redistribute
content.

## V1 Success

V1 succeeds when one agent can, using repository commands and documentation
alone:

- change report data or layout source;
- regenerate the HTML artifact;
- produce one readable image per page;
- inspect the images without connecting to a user's browser session;
- correct an obvious overflow or visual problem; and
- deliver a multi-page HTML report that opens without a running application.

A real report trial, not a toy title page, is required before the interface is
considered stable.

## Product Principles

- **Explicit pages.** Page membership is authored, not inferred.
- **Ordinary code.** Prefer familiar TypeScript, HTML, and CSS over a bespoke
  language.
- **Rendered truth.** Source is editable intent; browser output is the visual
  result.
- **Small interface.** Add a reusable concept only after real report work shows
  that it removes repetition or hides durable complexity.
- **Data stays ordinary.** Unslide receives ready-to-display values and does not
  own the caller's business calculations.
- **Agents use the same path.** Agent preview is a command over the real report,
  not a parallel mock renderer.
- **HTML stands alone.** Preview automation is not required to open the
  generated report.

## V1 Non-Goals

- Automatic pagination or content flow between pages.
- Animations, transitions, presenter controls, or slide navigation.
- A visual drag-and-drop editor.
- Automated report linting or a layout-rule system.
- Automatic overflow repair, font shrinking, or content truncation.
- A finance-specific data model, provenance system, or calculation engine.
- Cross-browser pixel parity or multiple rendering engines.
- A large design system or catalogue of speculative report primitives.
- PPTX, DOCX, or PDF as primary authoring formats.
- A stable public package interface before a real report has exercised it.

Browser printing or PDF capture may be convenient during development, but PDF
export is not a defining v1 capability.

## Beyond V1

Future work is accepted only in response to actual reports. Possible directions
include reusable themes, additional page sizes, PDF delivery, richer figures,
or automatic pagination. None is assumed by the initial architecture.
